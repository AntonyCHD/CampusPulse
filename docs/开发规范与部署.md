# 开发规范与部署

本文档从《详细设计.md》中拆分出来，只描述部署方式、开发工作流、编码规范、隐私规范和测试要求。

### 1. 部署方案

#### 1.1 本地开发模式

```bash
# 1. 创建环境
python -m venv .venv

# 2. 激活环境
# Windows
.venv\Scripts\activate
# macOS/Linux
source .venv/bin/activate

# 3. 安装依赖
pip install -r requirements.txt

# 4. 启动后端
uvicorn backend.app.main:app --reload --host 0.0.0.0 --port 8000

# 5. 启动前端
cd frontend-vue
npm install
npm run dev
```

#### 1.2 Docker Compose 部署

Docker Compose 官方定位就是用来定义和运行多容器应用，可以通过一个 YAML 文件管理服务、网络和卷，适合你们把前端、后端、数据库和向量库一起拉起来。([Docker Documentation][7])

`docker-compose.yml`：

```yaml
services:
  backend:
    build:
      context: .
      dockerfile: Dockerfile.backend
    ports:
      - "8000:8000"
    env_file:
      - .env
    volumes:
      - ./data:/app/data
      - ./cache:/app/cache
      - ./outputs:/app/outputs

  frontend:
    build:
      context: .
      dockerfile: Dockerfile.frontend
    ports:
      - "5173:5173"
    environment:
      - VITE_API_BASE_URL=http://backend:8000
    depends_on:
      - backend
```

启动：

```bash
docker compose up --build
```

### 3. 编码规范

#### 3.1 Python 规范

* Python 版本：3.11+
* 全部函数必须有类型标注。
* 业务对象必须用 Pydantic schema。
* 算法函数不要直接读文件。
* 服务层不要写 UI。
* API 层不要写算法。
* 大模型调用必须有超时、重试、缓存、schema 校验。

示例：

```python
def compute_resonance_score(comments: list[Comment]) -> float:
    """计算评论区情绪共振指数，返回 0-1 之间的分数。"""
    if not comments:
        return 0.0
    ...
```

#### 3.2 命名规范

```text
文件名：snake_case.py
类名：PascalCase
函数名：snake_case
变量名：snake_case
常量名：UPPER_CASE
API路径：kebab-case 或 snake_case，保持统一
```

#### 3.3 日志规范

日志至少记录：

```text
event_id
module
duration_ms
mode: realtime/cached
llm_model
cache_hit
error_type
```

示例：

```python
logger.info(
    "risk_assessment_finished",
    extra={
        "event_id": event.event_id,
        "risk_score": assessment.risk_score,
        "duration_ms": duration_ms,
        "mode": mode,
    },
)
```

#### 3.4 大模型调用规范

必须满足：

```text
1. 所有 prompt 放在 backend/app/llm/prompts/
2. 所有输出必须走 JSON Schema
3. 所有输出必须 Pydantic 校验
4. 校验失败走 fallback_template
5. 结果写入 cache/llm_responses/
6. 不允许在 prompt 中放真实个人信息
```

#### 3.5 前端规范

* 页面只调用 API，不直接跑算法。
* 图表数据由后端返回 JSON。
* 每个页面一个主任务。
* 所有异常都要展示为用户可读提示。
* 答辩用案例必须固定排序，避免现场找不到。

当前可运行前端采用 Vue 3 页面承载总览、单事件分析、对比实验、证据化处置和报告导出，不再使用 Streamlit 页面目录。

#### 3.6 数据与隐私规范

- `data/raw/` 只放本地原始授权数据，不提交 Git，不进入前端，不进入报告。
- 所有可展示数据只能来自 `data/processed/`、`data/demo_cases/`、`cache/demo_reports/`。
- 脱敏映射表如 `user_id -> user_hash` 只能临时存放在本地 `data/.private/`，不得进入 Git。
- 日志不得记录原始正文全文，只记录 `event_id`、评论数量、耗时、风险等级、错误类型。
- 报告里引用评论时只展示脱敏文本和评论编号。

#### 3.7 测试规范

MVP 至少要有以下测试：

```text
test_schema.py          校验 JSONL 输入和 Pydantic schema
test_anonymize.py       校验手机号、学号、宿舍号等敏感信息被替换
test_risk_scoring.py    校验风险分数范围、等级映射、空评论边界
test_comment_graph.py   校验 parent_id 边和时间边构建正确
test_analysis_api.py    校验 /api/analyze/{event_id} 返回字段完整
```

测试不追求覆盖所有算法效果，但必须防止三类错误：数据读不进来、隐私泄露、分析接口返回结构不稳定。
