## 1. 项目定位

项目全称建议：

**群声雷达：面向校园墙场景的舆情风险演化与证据化处置平台**

一句话：

> 基于授权校园墙数据，系统识别校园舆情事件从“吐槽、求证、情绪共振、立场对立、组织化行动、线下风险”的演化路径，并通过评论链建模、话题聚类、风险评分、RAG 证据检索与大模型结构化研判，生成可视化风险报告和温和处置建议。

这比初版更强。初版已经提出“主贴 + 评论区全链路对话链语义推理”，重点识别情绪共振、风险升级和线下行动号召，但技术路线仍然写成“不训练、不微调模型，只用 Prompt Engineering + 大模型 API + 领域知识注入”。现在要升级为：

**语义向量表示 + 规则兜底 + LLM 结构化弱标注 + 图结构分析 + RAG 证据检索 + 可视化 Demo。**

大模型是“结构化标注器”和“报告生成器”，不是整个系统的唯一大脑；真正的核心判断来自语义表示、评论链结构、风险信号和可解释评分。

## 2. 总体系统架构

推荐采用：

* **前端 Demo：Streamlit**
* **后端服务：FastAPI**
* **数据存储：SQLite / DuckDB**
* **语义表示：BGE-M3 / bge-large-zh-v1.5 / gte-Qwen2 系列 Embedding**
* **向量检索：FAISS / Chroma**
* **图谱计算：NetworkX**
* **大模型接口：OpenAI-compatible SDK，可切换 OpenAI、DeepSeek、通义、智谱**
* **可视化：Streamlit 原生图表 + Plotly / PyVis / ECharts JSON**
* **部署：本地运行 + Docker Compose 兜底**

FastAPI 适合作为后端 API 框架，因为它基于 Python 类型提示构建 API，并能自动生成交互式 API 文档；这对调试接口、验证后端不是脚本拼接很有帮助。([FastAPI][1]) Streamlit 适合快速做数据应用和 AI/ML Demo，官方定位就是用 Python 快速构建交互式数据应用。([Streamlit][2]) Pydantic 则负责数据校验，它基于 Python 类型提示定义数据结构，非常适合把“大模型输出 JSON”变成可验证对象。([Pydantic][3])

## 3. 功能模块设计

系统拆成 9 个模块。每个模块必须有明确输入、输出和验收标准。模块之间只传 Pydantic 对象或 JSON，不直接读写彼此内部变量。

| 模块 | 名称 | 设计层级 | 最小实现 | 输入 | 输出 | 验收标准 |
| --- | --- | --- | --- | --- | --- | --- |
| M1 | 数据导入与脱敏 | 必须 | 读取 JSONL，校验字段，脱敏文本，生成标准化事件文件 | `data/raw/*.jsonl` | `data/processed/events.jsonl` | 5-10 个事件可导入；缺字段报清晰错误；不输出真实姓名、手机号、学号、宿舍号、账号 ID |
| M2 | 数据标注 | 必须 | 支持人工维护标签文件，不做复杂标注平台 | `events.jsonl` + `labels.jsonl` | `LabeledEvent` | 每个演示事件至少有风险等级、阶段、关键评论、事件类型 |
| M3 | 话题聚类 | 增强 | 基础版可先一条 JSONL 代表一个事件；增强版再做 embedding 聚类 | `Post[]` | `EventCluster[]` | 基础闭环可跳过；不影响单事件分析闭环 |
| M4 | 评论链建模 | 必须 | 用主贴、评论、`parent_id`、时间顺序构建节点和边 | `CampusEvent` | `CommentGraph` | 每个评论都有节点；有 `parent_id` 用回复边，没有则按时间连边 |
| M5 | 风险信号识别 | 必须 | 语义向量 + 规则兜底 + LLM Top-N 结构化弱标注，识别求证、攻击、号召、隐私、谣言、不满等信号 | `CampusEvent` + `CommentEmbedding[]` | `RiskSignal[]` | 能解释每个信号来自哪条评论、哪段文本、哪类语义/规则/LLM 标注 |
| M6 | 风险演化评分 | 必须 | 计算风险分、等级、阶段、关键评论排序 | `RiskSignal[]` + `CommentGraph` | `RiskAssessment` | 单事件可稳定输出 `risk_score`、`risk_level`、`current_stage`、`key_comments` |
| M7 | RAG 证据检索 | 增强 | 基础版可返回空证据或手工模板；增强版再做向量检索 | `event_summary` | `Evidence[]` | 没有知识库时系统不崩；有知识库时能返回来源和片段 |
| M8 | LLM 结构化研判 | 核心增强 | 调用 OpenAI-compatible API 或读取缓存，输出受 schema 约束的报告 | `RiskAssessment` + `Evidence[]` | `FinalReport` | API 失败时可用 fallback 模板；输出必须通过 Pydantic 校验 |
| M9 | 可视化 Demo | 必须 | Streamlit 展示事件列表、单事件分析、关键评论、对比结果、缓存报告 | 后端 API JSON | UI 页面 | 无需改代码即可切换事件和缓存模式 |

### 3.1 模块依赖顺序

```text
M1 数据导入与脱敏
  -> M2 标注加载
  -> M4 评论链建模
  -> M5 风险信号识别
  -> M6 风险演化评分
  -> M8 LLM/模板报告
  -> M9 前端展示
```

M3 和 M7 是增强模块，不应阻塞基础分析闭环。基础闭环中不要把自动话题聚类、OCR、复杂 RAG、多智能体处置作为首要任务。

### 3.2 核心职责边界

- API 层只做请求解析、权限/模式参数、调用服务、返回 JSON。
- Service 层负责编排流程，例如 `AnalysisService.analyze_event(event_id)`。
- Algorithm 层只接收对象并返回计算结果，不直接读文件、不调用数据库、不调用 LLM。
- LLM 层只接收脱敏后的结构化上下文，不接收原始数据文件。
- Frontend 只消费 API 返回的数据，不直接导入算法模块。

核心思想：**语义表示、M4/M5/M6 是技术深度，M8 是解释和表达增强，不是替代算法。**

## 5. 数据输入与 JSONL 契约

当前数据集按 JSONL 准备：**一行是一个校园墙事件**，每行必须是独立合法 JSON。JSONL 的优点是便于增量追加、逐行校验、批量实验和脱敏处理。初始实现不要先上数据库采集器，先保证 `data/raw/campus_wall_sample.jsonl` 能稳定导入。

### 5.1 原始 JSONL 最小字段

文件位置：`data/raw/campus_wall_sample.jsonl`。原始文件不提交 Git，只在本地处理。

```json
{"event_id":"E001","post":{"post_id":"P001","text":"脱敏前主贴文本","created_at":"2026-06-20T21:30:00+08:00","like_count":12,"image_urls":[]},"comments":[{"comment_id":"C001","parent_id":null,"text":"脱敏前评论","created_at":"2026-06-20T21:35:00+08:00","like_count":23,"user_id":"raw_user_001"}]}
```

必填字段：

| 路径 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `event_id` | string | 是 | 事件编号。若原始数据没有事件概念，可用 `E` + 序号生成 |
| `post.post_id` | string | 是 | 主贴编号 |
| `post.text` | string | 是 | 主贴正文，允许为空格但导入时应清洗 |
| `post.created_at` | string | 是 | ISO 8601 时间，建议带 `+08:00` |
| `post.like_count` | integer | 否 | 缺失时置 0 |
| `post.image_urls` | string[] | 否 | 基础版可保留字段但不分析图片 |
| `comments[].comment_id` | string | 是 | 评论编号 |
| `comments[].parent_id` | string/null | 否 | 回复目标；没有回复结构时为 null |
| `comments[].text` | string | 是 | 评论正文 |
| `comments[].created_at` | string | 是 | 评论时间 |
| `comments[].like_count` | integer | 否 | 缺失时置 0 |
| `comments[].user_id` | string | 否 | 原始用户 ID，只能进入脱敏流程，不能进入 Demo 输出 |

### 5.2 标准化事件 JSONL

文件位置：`data/processed/events.jsonl`。这是后端和实验统一读取的文件。

```json
{
  "event_id": "E001",
  "event_type": "宿舍管理",
  "post": {
    "post_id": "P001",
    "source": "campus_wall",
    "user_hash": "U_8f31a9",
    "title": null,
    "text": "脱敏后的主贴内容",
    "clean_text": "清洗后的主贴内容",
    "created_at": "2026-06-20T21:30:00+08:00",
    "like_count": 12,
    "comment_count": 3,
    "image_paths": []
  },
  "comments": [
    {
      "comment_id": "C001",
      "parent_id": null,
      "user_hash": "U_45ab22",
      "text": "脱敏后的评论",
      "clean_text": "清洗后的评论",
      "created_at": "2026-06-20T21:35:00+08:00",
      "like_count": 23,
      "reply_count": 0,
      "emotion": null,
      "stance": null,
      "risk_signals": []
    }
  ],
  "tags": ["宿舍", "限电"],
  "data_note": "授权获取并脱敏处理"
}
```

标准化规则：

- 时间统一转为 ISO 8601 字符串；内部计算可转 `datetime`。
- `like_count`、`reply_count` 缺失时置 0，不能为负数。
- `comments` 允许为空，但用于完整分析的事件建议至少 10 条评论。
- `user_id`、头像、昵称、手机号、学号、宿舍号、真实姓名不得出现在 `processed`、`cache`、`outputs`。
- `text` 保留脱敏后原句；`clean_text` 用于算法，去除多余空白、URL、表情占位和重复标点。

### 5.3 人工标注 JSONL

文件位置：`data/labels/event_labels.jsonl`。人工标注用于对照评估、关键评论命中率计算和案例复核。

```json
{"event_id":"E001","risk_level":"高","risk_stage":"组织化行动","event_type":"宿舍管理","key_comments":["C007","C012"],"risk_reason":"评论区出现高赞线下询问/集合倾向","annotator":"A1","updated_at":"2026-06-23T20:00:00+08:00"}
```

标注规模建议：

| 用途 | 最低数量 | 推荐数量 | 说明 |
| --- | --- | --- | --- |
| Demo 演示 | 5-10 个事件 | 10 个事件 | 每个事件 10-30 条评论，覆盖低/中/高风险 |
| 论文初步实验 | 30 个事件 | 50-100 个事件 | 至少能计算准确率、召回率、关键评论命中率 |
| 中期增强 | 100 个事件以上 | 2000-5000 条评论 | 可尝试轻量分类器或更稳定的统计评估 |

### 5.4 分析输出 JSON

`POST /api/analyze/{event_id}` 的核心输出必须可缓存到 `cache/demo_reports/{event_id}.json`：

```json

  "risk_score": 83.5,
  "risk_level": "高",
  "current_stage": "组织化行动",
  "evolution_path": ["信息求证", "情绪共振", "组织化行动"],
  "risk_signals": [
```

## 8. 核心算法设计

这是整个项目最重要的部分。核心设计原则是：

**我们不是让大模型直接猜风险，而是先把评论区变成可计算的风险结构，再让大模型解释和生成建议。**

### 8.0 轻量语义建模路线

本项目不应把“传统 NLP”作为正式技术主线。正式路线采用更现代但仍轻量的语义建模：

```text
脱敏文本
-> 中文/多语义向量模型生成 embedding
-> 规则兜底识别明显高风险信号
-> LLM 只对 Top-N 关键评论做结构化弱标注
-> 评论链图谱、情绪共振指数、风险阶段和关键评论排序
-> LLM 基于结构化结果生成解释和处置建议
```

推荐模型选择：

| 模型/方法 | 用途 | 基础版是否建议 | 复杂度 |
| --- | --- | --- | --- |
| `BAAI/bge-m3` | 主贴/评论向量、语义相似度、RAG 检索 | 建议 | 中 |
| `bge-large-zh-v1.5` | 中文语义向量，资源占用相对可控 | 建议 | 中 |
| `gte-Qwen2` 系列 | 语义表示增强，适合后续升级 | 可选 | 中 |
| `sentence-transformers` | 本地批量编码和缓存 embedding | 建议 | 低 |
| LLM JSON 弱标注 | Top-N 评论情绪、立场、风险信号标注 | 建议缓存 | 低到中 |
| KMeans / HDBSCAN | 语义聚类和主题发现 | 增强 | 中 |
| 关键词/情感词典 | 只做兜底和基线对照 | 保留 | 低 |

工程约束：

- Embedding 结果必须缓存到 `cache/embeddings/` 或本地数据库，避免每次演示重复计算。
- LLM 只处理评论预过滤后的 Top-N，不全量扫评论，避免成本和不稳定性。
- 关键词和情感词典只作为 baseline 或兜底信号，不作为正式方法卖点。
- 论文表述应写“基于语义表示的评论链风险演化识别”，不要写成“传统 NLP 情感分析系统”。

### 8.1 数据脱敏算法

#### 目标

保护学生隐私，避免把真实姓名、手机号、学号、宿舍号、账号 ID 暴露到 Demo 和论文里。

#### 规则

* 姓名：替换为 `[姓名]`
* 学号：替换为 `[学号]`
* 手机号：替换为 `[电话]`
* 宿舍房间：替换为 `[宿舍号]`
* 用户 ID：hash 成 `U_xxxxx`
* 图片：只保留脱敏后截图，或 OCR 后仅存文本

#### 伪代码

```text
Algorithm AnonymizeEvent(event_raw):
    input: 原始校园墙事件
    output: 脱敏后的标准事件

    for each post/comment in event_raw:
        text = 原始文本
        text = 替换手机号为 [电话]
        text = 替换学号为 [学号]
        text = 替换宿舍号为 [宿舍号]
        text = 替换疑似姓名为 [姓名]
        user_id = hash(raw_user_id + salt)
        保存 clean_text 和 user_hash

    return anonymized_event
```

### 8.2 评论预过滤算法

初版已经提到“评论 >20 条时，按点赞、敏感词、互动筛 Top-N”。这点可以保留，但正式实现应升级为“互动强度 + 语义代表性 + 风险信号”的 Top-N 预过滤。

#### 目的

评论很多时，不把所有评论丢给大模型，而是先筛出最有代表性的评论，降低成本，提高解释性。

#### 评分公式

对每条评论计算：

```text
comment_importance =
    0.35 * normalized_like_count
  + 0.25 * semantic_representativeness
  + 0.20 * risk_signal_score
  + 0.10 * reply_depth_score
  + 0.10 * temporal_turning_score
```

#### 信号解释

* `like_count`：点赞越高，说明共鸣越强。
* `semantic_representativeness`：评论向量与事件中心向量越接近，越能代表主诉求。
* `risk_signal_score`：规则或 LLM 弱标注命中的行动号召、隐私泄露、传闻扩散等信号。
* `reply_depth_score`：被回复越多，说明引发讨论。
* `temporal_turning_score`：是否出现在风险急剧升高的时间段。

#### 伪代码

```text
Algorithm SelectTopComments(comments, top_n):
    for comment in comments:
        like_score = normalize(log(1 + comment.like_count))
        semantic_score = similarity(comment.embedding, event.centroid_embedding)
        risk_signal_score = estimate_risk_signal(comment)
        reply_score = normalize(comment.reply_count)
        turning_score = detect_temporal_turning(comment.created_at)

        comment.importance =
            0.35 * like_score +
            0.25 * semantic_score +
            0.20 * risk_signal_score +
            0.10 * reply_score +
            0.10 * turning_score

    sort comments by importance descending
    return top_n comments
```

### 8.3 话题聚类算法

#### 目的

校园墙同一事件可能出现多个帖子。比如：

* “今天宿舍又停电？”
* “为什么 12 点强制断电？”
* “限电影响期末复习怎么办？”

这些应该聚成一个事件，而不是三个孤立帖子。

#### 推荐实现

基础版：

* 用中文句向量模型生成 embedding；
* 用余弦相似度判断是否归为同一事件；
* 相似度超过阈值 `0.78` 归并。

增强版：

* 使用 BERTopic / HDBSCAN 聚类。BERTopic 使用 transformer embeddings 和 c-TF-IDF 形成可解释的话题簇，适合从大量文本中抽取主题。([Maarten Gr.github][5])

#### 伪代码

```text
Algorithm ClusterPosts(posts):
    embeddings = Encode(posts.clean_text)

    clusters = []
    for post in posts:
        best_cluster = None
        best_sim = 0

        for cluster in clusters:
            sim = cosine(post.embedding, cluster.centroid)
            if sim > best_sim:
                best_sim = sim
                best_cluster = cluster

        if best_sim >= TOPIC_THRESHOLD:
            assign post to best_cluster
            update cluster centroid
        else:
            create new cluster with post

    return clusters
```

### 8.4 评论链图谱构建算法

#### 目的

评论区不是一堆句子，而是一条会呼吸的河流。
我们要看清楚：

* 谁回复了谁；
* 哪条评论引发共鸣；
* 风险信号何时出现；
* 事件是降温还是升温。

NetworkX 适合做这部分，因为它是用于创建、操作和研究复杂网络结构、动态与功能的 Python 包。([NetworkX][6])

#### 图结构

节点：

* 主贴节点 `post`
* 评论节点 `comment`

边：

* `reply_edge`：回复关系
* `temporal_edge`：时间先后关系
* `semantic_edge`：语义相似关系

#### 节点属性

```json
{
  "risk_score": 0.72,
  "emotion": "愤怒",
  "stance": "号召",
  "like_count": 32,
  "created_at": "2026-06-23T20:30:00"
}
```

#### 伪代码

```text
Algorithm BuildCommentGraph(event):
    G = DirectedGraph()

    add node event.post_id as post node

    sort comments by created_at

    for comment in comments:
        add node comment.comment_id with attributes:
            emotion, stance, like_count, risk_signals

        if comment.parent_id exists:
            add edge parent_id -> comment_id, type="reply"
        else:
            add edge post_id -> comment_id, type="reply"

    for i from 0 to len(comments)-2:
        add edge comments[i] -> comments[i+1], type="temporal"

    for each pair(comment_i, comment_j):
        if cosine(embedding_i, embedding_j) > SEMANTIC_THRESHOLD:
            add edge i -> j, type="semantic", weight=similarity

    return G
```

### 8.5 风险信号识别算法

#### 风险信号类型

建议定义 8 类：

| 信号                   | 解释       | 示例         |
| -------------------- | -------- | ---------- |
| negative_emotion     | 明显负面情绪   | “真的受不了了”   |
| collective_resonance | 多人表达相同不满 | “我也是”“同感”  |
| rumor_spread         | 未证实传闻扩散  | “听说是因为……”  |
| sarcasm              | 反讽阴阳怪气   | “学校真是太贴心了” |
| mobilization         | 行动号召     | “今晚一起去问”   |
| confrontation        | 对抗倾向     | “不解决就闹大”   |
| privacy_leak         | 隐私泄露     | 曝光姓名、电话、宿舍 |
| offline_risk         | 线下聚集风险   | “去楼下集合”    |

#### 实现方式

基础版用“语义向量 + 规则兜底 + Top-N LLM 弱标注”：

* 中文语义向量模型生成 `comment.embedding`。
* 规则兜底识别明显风险词、隐私格式、时间地点和行动号召。
* LLM 只对 Top-N 评论做 JSON 弱标注，输出情绪、立场、风险信号和置信度。
* 关键词词典和情感词典只作为 baseline 或兜底，不作为主技术路线。

增强版：

* 人工标注 300-500 条评论。
* 训练 `embedding + Logistic Regression / LightGBM` 轻量分类器。
* 时间充足时再考虑微调中文 RoBERTa / MacBERT，不建议作为基础版。

#### 伪代码

```text
Algorithm DetectRiskSignals(comment, event_context):
    signals = []

    embedding = encode(comment.clean_text)
    comment.semantic_features = extract_semantic_features(embedding, event_context)

    if contains_negative_emotion(comment.text):
        signals.append("negative_emotion")

    if contains_collective_words(comment.text):
        signals.append("collective_resonance")

    if contains_uncertain_claim(comment.text):
        signals.append("rumor_spread")

    if contains_sarcasm_pattern(comment.text):
        signals.append("sarcasm")

    if contains_action_call(comment.text):
        signals.append("mobilization")

    if contains_privacy_pattern(comment.text):
        signals.append("privacy_leak")

    if contains_offline_location_and_time(comment.text):
        signals.append("offline_risk")

    return signals
```

### 8.6 情绪共振指数

这是你们论文最有“技术味”的指标之一。

#### 定义

情绪共振不是“负面评论多”这么简单，而是同时满足：

* 多人表达相同情绪；
* 时间上集中出现；
* 互动数据较高；
* 评论之间语义相似。

#### 公式

```text
ResonanceScore =
    α * negative_ratio
  + β * semantic_concentration
  + γ * interaction_amplification
  + δ * temporal_density
```

建议权重：

```text
α = 0.30
β = 0.25
γ = 0.25
δ = 0.20
```

#### 各项含义

```text
negative_ratio = 负向/焦虑/愤怒评论数 / 总评论数

semantic_concentration = 高相似评论对数量 / 全部评论对数量

interaction_amplification = 高点赞风险评论点赞数 / 总点赞数

temporal_density = 风险评论在短时间窗口内的集中程度
```

#### 伪代码

```text
Algorithm ComputeResonanceScore(comments):
    negative_ratio = count_negative(comments) / len(comments)

    semantic_pairs = 0
    high_sim_pairs = 0
    for each pair in comments:
        semantic_pairs += 1
        if cosine(pair.embedding) > 0.80:
            high_sim_pairs += 1
    semantic_concentration = high_sim_pairs / semantic_pairs

    risk_likes = sum(like_count of comments with risk_signals)
    total_likes = sum(all like_count) + 1
    interaction_amplification = risk_likes / total_likes

    temporal_density = max_risk_comments_in_window(comments, window=30min) / len(comments)

    score = 0.30*negative_ratio + 0.25*semantic_concentration
          + 0.25*interaction_amplification + 0.20*temporal_density

    return score
```

### 8.7 风险演化阶段识别

#### 阶段定义

```text
S0 个体吐槽：主贴负面，但评论稀疏
S1 信息求证：评论以“真的假的”“有通知吗”为主
S2 情绪共振：多人表达相同不满
S3 立场对立：评论区出现争吵、攻击、阵营化
S4 组织化行动：出现投诉、集合、转发扩散、线下行动号召
S5 线下风险：出现明确时间地点和行动目标
```

#### 阶段判断规则

| 阶段 | 触发条件              |
| -- | ----------------- |
| S0 | 主贴负面，评论少于 5 条     |
| S1 | 求证类评论比例 > 30%     |
| S2 | 共振指数 > 0.55       |
| S3 | 攻击/对立信号 > 20%     |
| S4 | 行动号召信号 ≥ 1 且互动较高  |
| S5 | 行动号召 + 时间地点 + 高互动 |

#### 伪代码

```text
Algorithm InferEvolutionStage(event, comments, signals):
    if len(comments) < 5 and post_is_negative(event.post):
        return "个体吐槽"

    if ratio(stance == "求证") > 0.30:
        stage = "信息求证"

    if resonance_score > 0.55:
        stage = "情绪共振"

    if confrontation_ratio > 0.20:
        stage = "立场对立"

    if has_signal("mobilization") and key_comment_like_high:
        stage = "组织化行动"

    if has_signal("offline_risk") and contains_time_location:
        stage = "线下风险"

    return stage
```

### 8.8 总风险评分算法

#### 风险评分公式

```text
RiskScore =
    20 * PostRisk
  + 25 * ResonanceScore
  + 20 * MobilizationScore
  + 15 * GraphInfluenceScore
  + 10 * BurstScore
  + 10 * EvidenceUncertainty
```

#### 解释

* `PostRisk`：主贴风险；
* `ResonanceScore`：评论区情绪共振；
* `MobilizationScore`：行动号召强度；
* `GraphInfluenceScore`：关键评论在图中的影响力；
* `BurstScore`：短时间突增；
* `EvidenceUncertainty`：事实不清时提高复核优先级。

#### 风险等级

```text
0-39   低
40-64  中
65-84  高
85-100 严重
```

#### 伪代码

```text
Algorithm ComputeRiskScore(event, graph, signals, evidence):
    post_risk = compute_post_risk(event.post)
    resonance = compute_resonance_score(event.comments)
    mobilization = compute_mobilization_score(signals)
    graph_influence = compute_key_node_influence(graph)
    burst = compute_burst_score(event.comments)
    uncertainty = compute_evidence_uncertainty(evidence)

    risk_score =
        20 * post_risk +
        25 * resonance +
        20 * mobilization +
        15 * graph_influence +
        10 * burst +
        10 * uncertainty

    risk_level = map_score_to_level(risk_score)
    return risk_score, risk_level
```

### 8.9 关键评论发现算法

#### 目的

系统不能只输出“风险高”，必须指出：

**是哪几条评论推动了风险升级。**

#### 特征

```text
comment_key_score =
    0.30 * risk_signal_score
  + 0.25 * graph_centrality
  + 0.20 * like_score
  + 0.15 * temporal_turning_score
  + 0.10 * semantic_representativeness
```

#### 伪代码

```text
Algorithm RankKeyComments(comments, graph):
    centrality = PageRank(graph)

    for comment in comments:
        risk_signal_score = count(comment.risk_signals) / MAX_SIGNAL_COUNT
        graph_score = centrality[comment.comment_id]
        like_score = normalize(log(1 + comment.like_count))
        turning_score = is_before_stage_shift(comment)
        semantic_score = similarity_to_event_centroid(comment)

        comment.key_score =
            0.30*risk_signal_score +
            0.25*graph_score +
            0.20*like_score +
            0.15*turning_score +
            0.10*semantic_score

    return top_k comments by key_score
```
